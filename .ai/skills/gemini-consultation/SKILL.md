# Gemini CLI 相談手順書

Gemini CLIにセカンドオピニオンを求め、その結果を要約してユーザーに報告する。
ユーザーはgeminiの生出力を直接見る必要はない。呼び出し側エージェント（Claude Code等）がgeminiの回答を咀嚼し、自分の見解と照らし合わせて報告する。

## 1. geminiインストール確認

Bashツールで以下を実行する:

```
command -v gemini
```

geminiが見つからない場合は、ユーザーに「Gemini CLIがインストールされていません」と報告して終了する。
それ以上の作業は行わない。

## 2. 相談の深さの確認

ユーザーが相談の深さを明示していない場合（「全力で」「深く」「軽く」「普通に」等の指定がない場合）、
AskUserQuestionツールで以下の3択を提示する:

| 選択肢                 | 往復回数                                          | 呼び出し側の仮見解          | 漏れ抜けの追求度             |
| ---------------------- | ------------------------------------------------- | --------------------------- | ---------------------------- |
| **全力で深く相談する** | 2往復を原則実施（明らかに不要な場合のみスキップ） | 必ず含め、geminiに反論を促す | 積極的に反論・補完・追加追求 |
| **よく相談する**       | 必要に応じて2往復                                 | 呼び出し側が判断            | 漏れ抜けを確認して報告       |
| **普通に相談する**     | 1往復のみ                                         | 呼び出し側が判断            | geminiの指摘を整理して報告   |

ユーザーが明示的に深さを指定している場合はこの確認を省略し、指定に従う。

## 3. プロンプトの設計

geminiに渡すプロンプトを呼び出し側が設計する。以下の指針に従う:

### プロンプト設計の指針

- 会話のコンテキストから「今何について相談すべきか」を判断する
- ユーザーが明示的にテーマを指定している場合はそれに従う
- 指定がない場合は、直前の議論や作業内容から適切なテーマを選ぶ

### コンテキスト別のプロンプト例

#### 未commitの変更をレビューしたいとき

「このリポジトリの未commitの変更をレビューしてください。コードの品質、設計の妥当性、潜在的なバグ、改善点を指摘してください。」

#### 設計について相談したいとき

「以下の設計について意見をください。[設計の要約]。代替案やトレードオフがあれば提案してください。」

#### 実装方針を相談したいとき

「[タスクの要約]を実装したい。このリポジトリのコードベースを踏まえて、最適な実装方針を提案してください。」

### プロンプトの構成

1. プロジェクトの背景（簡潔に）
2. 相談したい具体的な内容
3. 呼び出し側の現時点での仮見解（状況に応じて含める。「全力」モードでは必ず含める）
4. 指定観点以外も幅広く探索するよう依頼（常に含める）

#### 呼び出し側の仮見解を含める場合のテンプレート

```
なお私（呼び出し側）は現時点で以下のように考えている:
- [仮見解]

この見解の漏れ・誤りも含めて、幅広くレビューしてほしい。
指定した観点以外にも気になる点があれば積極的に報告してほしい。
```

#### 幅広い探索の依頼（常にプロンプトに含める）

指定された観点はあくまで出発点であり、制約ではない。
リポジトリを自分で探索して気になる点があれば積極的に報告してほしい、という旨を必ずプロンプトに含める。

## 4. geminiの実行

Bashツールで以下の形式で実行する:

```
gemini -p "ここにプロンプトを入れる" --approval-mode yolo
```

### 重要なオプション

- `-p` または `--prompt` は必須。非対話モードで実行する
- `--approval-mode yolo` は非対話実行（`-p`）でハング回避のため付与する。**yoloは全自動承認モードで、書き込みや破壊的操作も通過する**。相談（分析）用途では理想的には読み取り限定モードを使いたいが、現状 Gemini CLI に該当モードがないため yolo を使用する。副作用のリスクを理解した上で使用する

### タイムアウト

geminiの実行には時間がかかる場合がある。Bashツールのtimeoutを600000（10分、`BASH_MAX_TIMEOUT_MS` の上限）に設定する。

### 追加往復の判断

「全力」モードでは、1往復目の結果を受け取った後、2往復目を原則実施する。以下の**すべて**に該当する場合のみスキップする:

- 指摘が全て具体的・実装可能で、呼び出し側が追加の疑問・反論を持たない
- 1往復目で相談範囲が網羅されている（未探索領域が呼び出し側視点で特定できない）
- 呼び出し側の仮見解に対するgeminiの検証が完結している

「よく」モードでは、1往復目の結果を受け取った後、以下のいずれかに該当する場合は2往復目を実施する:

- さらに深掘りすべき重要なポイントがgeminiの指摘に含まれている
- geminiの指摘に呼び出し側が反論・補完したいことがある
- 十分に探索されていない領域があると呼び出し側が判断できる

2往復目を判断する前に、1往復目の出力にコマンド失敗の兆候がないか確認する。失敗を検知した場合は、まず呼び出し側が正しい情報を取得・補正してから2往復目の要否を判断する（詳細はセクション5「geminiのコマンド失敗を検知した場合」を参照）。

2往復目のプロンプト構成:

**注意: `gemini -p`は単発実行のため、2往復目のgeminiは1往復目の記憶を持っていない。最初から文脈を全て含める必要がある。**

1. 元の相談内容の再提示（背景/目的/制約/評価観点を落とさず構造化して含める）
2. 1往復目のgeminiの回答サマリー
3. 呼び出し側の反論・補完・追加質問
4. さらに掘り下げてほしい観点

往復は原則2回まで。3回以上はユーザーに確認してから実施する。

## 5. 結果の要約と報告

geminiの出力を読み取り、以下の構成でユーザーに報告する:

### 報告の構成

#### geminiへの相談内容

geminiに送ったプロンプトの概要を1-2文で説明する。

#### geminiの回答サマリー

geminiの主な指摘・提案を箇条書きで要約する。

#### 呼び出し側の見解

- geminiの指摘のうち、呼び出し側も同意するもの
- geminiの指摘のうち、呼び出し側が異なる意見を持つもの（理由付き）
- geminiが見落としていると呼び出し側が考えるポイント
- 2往復目を実施した場合は、その理由と2往復目で何が明らかになったか

**見解の姿勢:**

- geminiの指摘を安易に受け入れない。呼び出し側として検証を試みてから報告する
- geminiが探索しなかった可能性がある領域を意識的に探す
- 呼び出し側と、呼び出されたgeminiの見解が食い違う場合は、その食い違いを価値ある情報として積極的に伝える

#### 次のアクション提案

ユーザーと相談すべき事項や、推奨する次のステップを示す。

### 報告のスタイル

- 簡潔にまとめる。冗長な引用は避ける
- 呼び出し側と呼び出された側の見解が異なる場合は、両方の理由を説明してユーザーが判断できるようにする
- ユーザーが次に何をすべきか明確にする

### geminiの出力が大きすぎる場合

geminiの出力量が多く、要約や処理が困難だと判断した場合は、以下の手順に従う:

1. まずユーザーに「geminiの出力が非常に長かった」旨を報告する
2. 出力の冒頭部分から読み取れる概要だけを伝える
3. ユーザーと相談して対処方法を決める（例: 観点を絞って再実行、プロンプトに出力量の制約を追加、など）

事前にプロンプトで出力量を制限しない。制限すると重要な情報が欠落するリスクがある。

### geminiのコマンド失敗を検知した場合

geminiの出力を読む際、コマンド実行の失敗の兆候がないか注意深く確認する。
兆候の例: エラーメッセージ、「取得できなかった」「失敗した」「permission denied」「not found」など。

失敗を検知した場合は、以下の手順に従う:

1. まず、呼び出し側が正しい情報を自分で取得する。geminiが失敗したコマンドをBashツールで実行し、結果を確認する
2. 次に、取得した情報でgeminiの分析を補正する。情報不足で的外れになった指摘を特定し、正しい文脈で再評価する
3. 最後に、報告の「呼び出し側の見解」で情報不足を明示する。欠落情報と、その影響を受けた指摘を具体的に説明する

## 6. スキルの自己改善

geminiが何らかの失敗をしたパターンを認識したら、次回以降の改善策をユーザーに進言する。
