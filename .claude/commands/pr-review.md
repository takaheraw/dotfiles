## PR Review

Pull Request の体系的レビュー。

### レビューフロー

1. **事前確認**: `gh pr view` で PR 情報・関連 Issue を確認、`gh pr diff` で変更差分を取得
2. **体系的チェック**: セキュリティ → 正確性 → パフォーマンス → アーキテクチャの順で確認
3. **フィードバック**: 具体的な改善案とコード例を添えてコメント
4. **フォローアップ**: 修正確認、CI 状態チェック

### レビュー観点

- **セキュリティ**: 認証・認可、入力検証（SQLi/XSS）、機密情報の取り扱い
- **正確性**: ロジックエラー、境界値、Null チェック、エラーハンドリング
- **パフォーマンス**: N+1 クエリ、計算量、メモリリーク、キャッシュ
- **アーキテクチャ**: レイヤー分離、結合度、SOLID 原則

### コメント分類

| 重要度 | ラベル | 対象 |
|--------|--------|------|
| 致命的 | `critical.must` | セキュリティ脆弱性、データ整合性、システム障害リスク |
| 高 | `high.imo` | 機能不全リスク、パフォーマンス問題、保守性低下 |
| 中 | `medium.imo` | 可読性、コード構造、テスト品質 |
| 低 | `low.nits` | スタイル統一、タイポ、コメント |
| 質問 | `info.q` | 実装意図の確認、設計判断の背景 |

### コメント形式

```
{ラベル} {問題の説明}

改善案:
{具体的なコード例}

{理由・効果}
```

### 制約

- 建設的なトーンで、問題指摘には必ず改善案を添える
- Critical → High → Medium → Low の優先度順で対応を求める
