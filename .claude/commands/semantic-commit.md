## Semantic Commit

大きな変更を意味のある最小単位に分割し、セマンティックなコミットメッセージで順次コミットする。

### 動作フロー

1. **変更分析**: `git diff HEAD` で全変更を取得
2. **グループ化**: 変更ファイルを論理的な単位に分割
3. **順次コミット**: 各グループを `git add <files> && git commit` で順次実行
4. **確認**: 未コミットの変更が残っていないか検証

### 分割の基準

1. **機能単位**: 同一機能に関連するファイル（例: `src/auth/` 配下をまとめる）
2. **変更種別**: テスト、ドキュメント、設定ファイルなど同種の変更をまとめる
3. **依存関係**: モデル + マイグレーション、コンポーネント + スタイルなど相互に関連するファイル
4. **変更規模**: 1 コミットあたり 10 ファイル以下を目安

### Conventional Commits 形式

```
<type>[optional scope]: <description>
```

**タイプ**:
- `feat`: 新機能 / `fix`: バグ修正
- `docs`: ドキュメント / `test`: テスト
- `refactor`: リファクタリング / `perf`: パフォーマンス改善
- `chore`: その他 / `ci`: CI 設定 / `build`: ビルド / `style`: フォーマット

**Breaking Change**: `feat!: description` または footer に `BREAKING CHANGE:`

### プロジェクト規約の自動検出

以下の順で規約を検出し、最優先で従う：

1. **CommitLint 設定**: `commitlint.config.*` / `.commitlintrc.*` / `package.json` の `commitlint` セクション
2. **既存コミット履歴**: `git log --oneline -50` からタイプ・言語のパターンを分析
3. **Conventional Commits 標準**: 上記が見つからない場合のフォールバック

### 言語判定

コミットメッセージの言語を以下の順で判定：

1. CommitLint 設定で `subject-case` が無効化 → 日本語
2. 直近 20 コミットの 50% 以上が日本語 → 日本語
3. その他 → 英語

`--lang` オプションで強制指定も可能。

### 制約

- 自動プッシュは行わない
- 現在のブランチでコミット（ブランチ作成なし）
- プリコミットフック失敗時は自動修正を取り込んでリトライ（最大 2 回）
